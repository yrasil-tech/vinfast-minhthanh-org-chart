## Required environment (`.env.local`)

- `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` — used by `lib/supabase.ts`
- `EDIT_PASSWORD` — compared in `app/api/auth/route.ts`; falls back to a hard-coded default if unset

The Supabase schema and seed data live in `supabase/schema.sql` (one table: `org_nodes`, ~71 rows). Run it in Supabase SQL Editor to initialize a new project.

## Architecture

Single-page Next.js 14 (App Router) app in Vietnamese that renders an editable org chart. All UI is one client route (`app/page.tsx`) backed by two API routes.

### Data flow

- **Table**: `org_nodes` — flat rows joined into a tree via `parent_id` (empty string `''` marks the root, not `NULL`). Ordered in UI by `sort_order`.
- **Read path**: `GET /api/nodes` → `lib/supabase.ts` → client stores flat array in `page.tsx` state and also passes to `OrgChart`.
- **Write path**: `POST/PUT/DELETE /api/nodes` — each handler re-checks the `org_auth` cookie. DELETE walks `parent_id` server-side to cascade-delete the entire subtree and returns the deleted id list. POST auto-generates ids in the form `N{zero-padded}` from max existing id when none is supplied.
- **Supabase RLS is permissive** (public select/insert/update/delete). Write authorization is enforced only by the Next.js API routes via the `org_auth` cookie, so do not call Supabase directly from the client for mutations.

### Auth

- `POST /api/auth` sets an `httpOnly` cookie `org_auth=authenticated` (7-day maxAge) when the password matches `EDIT_PASSWORD`.
- `GET /api/auth` reports auth state to the client; `DELETE` clears the cookie.
- API mutation handlers call `isAuth(req)` — reject with 401 if the cookie is missing.

### OrgChart rendering — imperative, no re-render on edit

`components/OrgChart.tsx` wraps `d3-org-chart` and is dynamically imported with `ssr: false` (it needs the DOM). It initializes **once** when `initialData.length > 0` and exposes an imperative handle (`OrgChartHandle`) via `forwardRef`:

- `updateNode(node)` — does **not** call `chart.updateNode`. Instead it mutates the chart's internal `getChartState().data` / `allNodes`, then does a direct `outerHTML` swap on `[data-node-id="<id>"]`. This preserves expand/collapse state and avoids the full tree rebuild that `d3-org-chart` triggers on data changes.
- `addNode` / `removeNode` — delegate to `chart.addNode` / `chart.removeNode` (these preserve expand state).
- `fit` / `expandAll` / `collapseAll` / `zoomIn` / `zoomOut` — chart controls.

`app/page.tsx` uses these via `chartRef.current?.updateNode(...)` after a successful save. Parent React state (`nodes`) is kept in sync **only for header stats** (`filledCount`), not to re-drive the chart.

Node HTML is generated by `buildNodeContent()` which embeds inline styles (`dept.bg`, `dept.border`) plus inline `onclick`/`ondblclick`/`onmouseover` handlers. Edit clicks bubble up through `window.__onNodeEdit` (wired in a `useEffect`) because the HTML is rendered outside React. When changing node rendering, keep the `data-node-id` attribute — `updateNode` relies on it for DOM lookup.

### Types & departments

`lib/types.ts` defines the `Department` union (`leadership` | `sales` | `finance` | `hr` | `service`) and the `DEPT_CONFIG` color palette used by both `OrgChart` and `NodeEditor`. The `dept_color` column on `org_nodes` duplicates `DEPT_CONFIG[dept].border`; keep them aligned when adding or recoloring departments (see `handleDeptChange` in `NodeEditor.tsx`).

`types/d3-org-chart.d.ts` is a local ambient type stub — `d3-org-chart` ships without TypeScript types.

### Path alias

`tsconfig.json` maps `@/*` → repo root (used as `@/lib/...`, `@/components/...`).
